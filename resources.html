01 Context

① framing

เปิดโปรเจกต์ + สั่งร่าง CONTEXT.md

เริ่มโปรเจกต์ใหม่: ระบบฐานข้อมูลพนักงาน (web app + REST API + PostgreSQL) ยังไม่มีโค้ด
ช่วยร่าง CONTEXT.md แล้ว mirror เป็น CLAUDE.md/AGENTS.md
— อย่าเพิ่งเขียนโค้ด feature ถ้าไม่ชัดให้ถามก่อน ห้ามเดา
② structure

กำหนดหัวข้อที่ context ต้องมี

ขอให้ CONTEXT.md มีหัวข้อครบ:
- Hard rules (บนสุด)
- Project + โดเมน
- Stack (+ version)
- Commands (dev / test / lint / db)
- Structure (tree + layer)
- Conventions (naming / error / response)
- Database (schema / migration / no raw SQL)
- API & Security (validation / auth / secret)
- Testing

ใช้ตาราง + tree แทนพารากราฟ, term/ชื่อไฟล์เป็นอังกฤษ, กฎเจาะจง + ตรวจได้

02 Spec

spec ชัด

feature: เพิ่มพนักงานใหม่

HR เพิ่มพนักงานใหม่ด้วย name / email / department / position / startDate / status
- email ซ้ำ → 409
- field ว่าง → error ใต้ช่อง
- สำเร็จ → กลับ list เห็นทันที
- มี test: happy + invalid + duplicate

03 Plan

plan request

ขอแผนเป็นก้าวเล็ก ๆ ก่อน

เสนอแผนก่อน อย่าเพิ่งเขียนโค้ด
- แตกเป็นก้าวเล็ก: DB → validation → API → UI → tests
- ระบุไฟล์ที่จะแตะแต่ละก้าว + Definition of Done
- ถ้ามีจุดคลุมเครือ (เช่น กติกา/สิทธิ์/รูปแบบข้อมูล) ให้ถามก่อน ห้ามเดา

04 Implement

kickoff

อ่าน @CONTEXT.md + แผนนี้ แล้วลงมือ step 1 (Prisma model + migration) ทีละก้อน
- ทำตาม recipe ใน CONTEXT.md

05 Vibe QA

test · ประกบ AC

เขียน test (อ้างอิง @tests/ เดิม)

เขียน test ให้ feature "เพิ่มพนักงาน"
อ้างอิง AC ใน @CONTEXT.md
ทำตาม pattern เดิมใน @tests/ (โครงสร้าง · การตั้งชื่อ · helper เหมือนของเดิม)

ครอบ 2 ระดับ:
- API: POST /employees → 201 / 409 / 400 / 403
- Scenario e2e: กรอกฟอร์ม → เห็นใน list → แก้ได้

ทุก AC ต้องมี test ประกบ + เคสพัง (ว่าง · ซ้ำ · สิทธิ์)
ใช้ framework เดิมในโปรเจกต์
ห้ามแก้ business logic เพื่อให้ test ผ่าน
context file

เพิ่มส่วน Testing ลง CONTEXT.md

อ่าน @tests/ แล้วสรุปเป็นกฎลง CONTEXT.md:
- framework + ที่อยู่ test + วิธีรัน
- naming + โครงสร้าง (table-driven?)
- ระดับที่ต้องมี: API + scenario (e2e)
- กฎ: ทุก AC ต้องมี test ประกบ
- helper/fixture ที่ reuse ได้

07 Workshop · CPMatch Care

① kickoff · context

prompt ตั้งโปรเจกต์ตัวเต็ม

ต้องการสร้างโปรเจกต์ใหม่ "CPMatch Care" — แอพสำหรับพนักงานใช้ในองค์กร นี่คือรายละเอียด:

ฟีเจอร์ (ภาพรวม):
- ข้อมูลพนักงานทั้งหมดในองค์กร
- ฟังก์ชันขาด / ลา / มาสาย
- จัดการคนเข้า project ทำได้โดย admin
- user level หลายระดับ แต่ละระดับมีสิทธิ์ไม่เท่ากัน ตั้งได้อิสระ
- ระบบให้แต้มเมื่อทำกิจกรรมที่ระบุไว้สำเร็จ
- กล่องระบายความรู้สึกแบบไม่ระบุตัวตน + ใช้ AI สรุปให้หัวหน้าอ่าน
- หน้า admin จัดการระบบ (อยู่ในส่วน code ของ web)

นี่เป็นภาพรวมคร่าว ๆ จะลงรายละเอียดตอนทำ feature อีกครั้ง
ยังไม่ต้อง implement — ให้แพลนให้เสร็จและทำทีละ feature

โครงสร้างไฟล์ context:
- Layer 1: boilerplate สำหรับ agent หลายยี่ห้อ (agent.md, claude.md, gemini.md) ระบุด้วยว่า "ห้าม commit git ถ้าไม่มีการสั่ง"
- Layer 2: context.md = architecture ทั้งหมดของระบบ
- Layer 3: plan ทุก feature ต้องระบุสถานะ implement: todo / implement / ready to test / done เสมอ

รายละเอียดเชิงเทคนิค:
- มีทั้ง website และ mobile app
- Web: Next.js + Tailwind, ดึงข้อมูลผ่าน API ด้วย React Query (TanStack Query) — ไม่ต่อ DB ตรง (แนะนำการวาง structure ด้วย)
- App: Flutter เป็นหลัก (แนะนำ stack อื่น + structure ด้วย)
- API: Go + GORM, raw query ให้น้อยที่สุด, วาง folder แบบ Uncle Bob clean architecture
- จัดการ schema/migration ด้วย GORM AutoMigrate (ไม่ใช้ migration tool แยก)
- DB: Postgres + สร้าง docker ให้ด้วย
- ทุก service ก่อนจบ plan ต้องสรุป tech stack + architecture design ให้ครบก่อนเริ่มลงมือ
- unit test ที่ API + UI test ผ่าน Playwright, coverage > 80%

โครงสร้างไฟล์ระบบ:
- root เก็บ context.md และ claude.md ทั้งหมด
- แต่ละ service แยก git submodule: /api /web /app ภายใต้ root เดียวกัน
- /plans เก็บ plan แต่ละ feature เป็น 001-planname.md, feature ใหม่เพิ่ม plan ในนี้เสมอ และทุก plan ref ไปที่ context.md หลัก

skills ที่ต้องการ (ช่วยตรวจสอบ + install ก่อนเริ่ม):
- /gstack — https://github.com/garrytan/gstack
- /ux-ui-pro-max — https://ui-ux-pro-max-skill.nextlevelbuilder.io
- /impeccable — https://impeccable.style

ช่วยสร้าง role.md ระบุ persona หลายตำแหน่งไว้เป็นทีมที่ปรึกษาสำหรับ feature ใหม่
ขั้นต้นทีมเล็ก: uxui 1 · dev lead 1 · po 2 · พนักงาน (end-user) 1
เมื่อขึ้น plan feature ใหม่ ให้ถามทุกครั้งว่าจะใช้ทีมนี้หรือไม่

จบขั้นตอนนี้ ช่วยเขียน context.md สรุปให้ก่อน หากมีอะไรให้ถามเพิ่มเติม
สร้าง context ให้เสร็จก่อนค่อยสร้าง plan ต่อ
